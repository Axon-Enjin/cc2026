# Create & Conquer 2026 Hackathon Guide

This document turns the participant primer into a practical working reference for teams building during the hackathon.

## 1. Event Snapshot

- Event: Create & Conquer 2026 Hackathon
- Organizer: FEU Institute of Technology Computer Engineering Organization, with FEU Innovation Center and ICpEP.SE-NCR
- Audience: JHS, SHS, and undergraduate students in the Philippines
- Team size: 4 to 6 members
- Modality: Face-to-face in Metro Manila, with an online option for remote participants
- Prize pool: At least PHP 18,000 total, subject to change

## 2. Core Dates

- Registration opens: June 9, 2026
- Registration closes: June 20, 2026, 11:59 PM
- Confirmation emails end: June 22, 2026
- Orientation and kickoff: June 23, 2026, 8:00 AM to 4:00 PM
- Proposal deadline: June 26, 2026, 1:00 PM
- Finalists announced: June 27, 2026, 1:00 PM
- Final output submission: June 29, 2026, 11:59 PM

## 3. What Teams Can Build

The hackathon welcomes a broad range of outputs:

- Mobile applications
- Web applications
- Games
- Web extensions
- Websites
- IoT projects
- Hardware projects
- Robots
- Any other technology-based output

No coding experience is required. Strong ideas, design, product thinking, and presentation can still compete well.

## 4. Eligibility and Registration

- Participants must be students in the Philippines.
- Mixed schools, courses, and universities are allowed.
- Each person may join only one team.
- If a participant is under 18, a parent or legal guardian consent form is required.
- Team leaders are responsible for registering the team.
- Registration is handled through the official form shared by the organizers.

## 5. Event Rules That Matter

- Work must begin during the hackathon. Pre-existing projects are not allowed as submissions.
- Pre-existing assets from personal projects may be allowed if they are only supporting pieces, not the full project.
- Reusing a pre-existing project, even with approval, can disqualify a team from finalist consideration.
- For code entries, version control is mandatory and the repository should show real progress.
- Participants must follow venue rules and safety requirements.
- Teams own the work they develop during the event.
- Organizers may use submitted images and materials for event purposes.
- Information shared during the hackathon is at the participant’s risk, so sensitive data should be protected.

## 6. Competition Flow

### Orientation and Kickoff

The kickoff introduces the contest rules, judges, mentors, submission process, and the hackathon theme. The theme is revealed during the kickoff.

Practical notes:

- At least one person from each team must attend.
- Attendance is checked at the start.
- Online attendees should keep mic and video usage respectful and limited to what is needed.
- In-person attendees may use the venue workspace after kickoff.

### Elimination Round

Teams must submit:

- Proof of concept
- Project brief
- Short video demo or presentation, 1 to 3 minutes
- Optional GitHub link for code projects

Top 8 teams advance to the finals.

### Final Round

Finalists present:

- 5 minute pitch
- 15 minute Q&A
- Finalized project brief
- Finalized proof of concept
- Presentation deck
- 30 second teaser video
- GitHub repository for code teams
- Deployed prototype if available, as a bonus

Teams may improve their work after elimination round submission, but the overall idea must remain the same.

## 7. Submission Checklist

### Elimination Round

Required for all teams:

- Proof of concept: user flow, wireframe, lo-fi UI prototype, sketch, or model
- Project brief
- 1 to 3 minute video demo or presentation

Optional for code teams:

- Public GitHub repository
- README with how-to-use instructions

### Final Round

Required for all teams:

- Finalized proof of concept
- Finalized project brief
- Presentation deck in PDF
- 30 second project teaser video

Required for code teams:

- Public GitHub repository
- README with how-to-use instructions

Optional for code teams:

- Deployed prototype link

## 8. What the Project Brief Should Include

The project brief should clearly cover:

- Project name
- Problem statement
- Solution and innovation
- Features and components
- Proof of concept
- Prototype
- Team members and roles
- References
- AI tools used in development

Optional but helpful sections:

- Market segment
- TAM, SAM, SOM
- Business model
- Target customer
- Channels
- Revenue streams
- Cost structure
- Partners
- Project persona

## 9. Judging Criteria

### Elimination Round

- Theme relevance: 10%
- Innovation and creativity: 30%
- Impact: 10%
- Execution: 30%
- Relevance and feasibility: 20%

### Final Round

- Theme relevance: 10%
- Innovation and creativity: 30%
- Impact: 10%
- Execution and presentation quality: 30%
- Relevance and feasibility: 20%

## 10. How to Build for This Hackathon

A strong team should focus on the following in order:

1. Understand the revealed theme as a problem to solve, not just a label.
2. Define one clear user and one clear outcome.
3. Build a simple proof of concept early, even if it is just a wireframe or lo-fi prototype.
4. Keep the project brief updated while building.
5. If coding, keep the repository public and commit progress regularly.
6. Prepare a short, clear demo video that shows the problem, solution, and value quickly.
7. Make sure every deliverable matches the same project story.

## 11. Suggested Team Roles

A 4 to 6 person team can usually split work like this:

- Product lead: keeps scope, story, and timeline on track
- Designer: handles wireframes, UI, and presentation visuals
- Developer 1: front-end or prototype implementation
- Developer 2: back-end, logic, or integrations
- Presenter: owns demo flow and Q&A preparation
- Research/documentation lead: manages brief, citations, and submission files

Some teams may combine roles depending on size.

## 12. Useful Building Targets

When deciding what to make, the judges will look for a project that is:

- Clearly tied to the theme
- Original enough to feel specific, not generic
- Realistic to build or extend
- Easy to explain in a short pitch
- Supported by a thoughtful prototype
- Documented well enough to understand quickly

## 15. Working Note For Teams

This hackathon rewards clarity as much as polish. A good submission should show:

- A real problem
- A believable solution
- A clean proof of concept
- A brief that matches the prototype
- A demo that is easy to follow

If the team keeps those five things aligned, the final submission will be much easier to defend during judging.
